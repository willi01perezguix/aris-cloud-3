# ADR-001: Arranque estructurado v1.1.0 con estabilidad v1.0.5

- Estado: Aprobado (Day 1)
- Fecha: 2026-02-14
- Decisores: Equipo ARIS_CONTROL_2

## Problema actual
El sistema cerró `v1.0.5` estable, pero la siguiente etapa (`v1.1.0`) necesita cambios estructurados de alto valor sin romper operación ni introducir cambios silenciosos de contrato API.

## Opciones evaluadas
1. **Big-bang en una sola rama/PR grande**
   - Pros: menor overhead documental.
   - Contras: alto riesgo de regresión y rollback complejo.
2. **Tracks paralelos con PRs pequeños + feature toggles**
   - Pros: control de riesgo, rollback parcial, visibilidad por objetivo.
   - Contras: coordinación adicional.
3. **Congelar desarrollo y solo hardening**
   - Pros: mínima exposición.
   - Contras: bajo avance de valor funcional.

## Decisión tomada
Adoptar **tracks paralelos (A-D)** con **PR pequeño por objetivo**, usando **feature toggle** cuando una decisión afecte UX crítica o permisos sensibles, y versionado explícito para cambios de contrato API.

## Impacto en UI / SDK / Backend
- UI: mejoras operativas sin contrato (badge global de salud + atajos de diagnóstico/incidencias).
- SDK: sin cambios de contrato en Day 1.
- Backend: sin cambios de endpoints/payload en Day 1.

## Plan de rollback
1. Revertir PR individual del objetivo afectado.
2. Desactivar feature toggle asociado (si aplica).
3. Ejecutar smoke corto Tenant/Store/User.
4. Registrar incidente y evidencia en paquete de soporte.

## Decisiones que requieren feature toggle
- Ajustes de UX que alteren navegación principal.
- Endurecimiento de permisos con riesgo de falso positivo en operadores.
- Cualquier comportamiento que degrade disponibilidad percibida de módulos core.
