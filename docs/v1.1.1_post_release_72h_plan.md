# Plan de monitoreo post-release v1.1.1 (72h)

## Objetivo
Monitorear estabilidad técnica y adopción de SDK `v1.1.1` durante las primeras 72h tras publicación estable.

## Cadencia operativa

### T+0 (publicación)
- Verificar disponibilidad de artefactos (`sdist`/`wheel`) en canal de distribución.
- Ejecutar smoke rápido de instalación en entorno controlado.
- Confirmar que no hay errores de import/config strict reportados.

### T+24
- Revisar tasa de fallos de inicialización SDK por versión de Python (enfasis 3.10/3.11).
- Revisar errores de polling exports (`FAILED` terminal vs `READY`) y latencias p95/p99.
- Validar que no hay incremento de retries agresivos por configuración de jitter.

### T+48
- Auditoría de incidents abiertos/cerrados relacionados con v1.1.1.
- Comparar baseline vs release previa (`v1.1.0`) en:
  - errores de config,
  - tiempos de resolución de export,
  - incidencias de cache inesperada.

### T+72
- Cierre de ventana de observación inicial.
- Decisión: mantener `v1.1.1` como recomendada o emitir parche.
- Publicar resumen de salud post-release.

## Métricas a vigilar
1. **Init/Import health**
   - `% de sesiones con import exitoso`
   - `% de errores por dependencia/config`
2. **Config strict health**
   - Conteo de `ConfigError` por ausencia de `ARIS3_API_BASE_URL`
3. **Exports polling health**
   - Latencia `READY` p50/p95/p99
   - Tasa de estados terminales `FAILED`
4. **Retry behavior health**
   - Reintentos promedio por request idempotente
   - Distribución de delays cuando jitter está activo
5. **Calidad de soporte**
   - tickets abiertos v1.1.1 / 24h
   - MTTR de incidentes asociados

## Umbrales de alerta
- Import/init failures > **1.0%** en cualquier ventana de 6h.
- `ConfigError` > **5 casos/h** sostenidos por 2h.
- `FAILED` terminal en exports > **3%** por 6h.
- p95 de `READY` aumenta > **30%** contra baseline de `v1.1.0`.
- Tickets críticos (sev1/sev2) >= **2** en 24h.

## Responsables y acciones
- **Release owner (SDK):** monitoreo diario, decisión de contención/rollback.
- **Soporte técnico:** triage de tickets y recolección de reproducibilidad.
- **Backend/API owner:** correlación con salud de API y mitigaciones de infraestructura.

### Acciones por nivel
- **Alerta amarilla (warning):** abrir incidente, activar seguimiento cada 4h.
- **Alerta roja (critical):** war-room, congelar despliegues dependientes y evaluar rollback inmediato a `v1.1.0`.
